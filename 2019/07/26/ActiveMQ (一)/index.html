ActiveMQ（一）

Posted on 2019-07-26

背景

ActiveMQ消息中间件是Apache所提供的一个开源的消息系统，完全采用Java来实现，因此，它能很好地支持J2EE提出的JMS（Java Message Service,即Java消息服务）规范。JMS是一组Java应用程序接口，它提供消息的创建、发送、读取等一系列服务。JMS提供了一组公共应用程序接口和响应的语法，类似于Java数据库的统一访问接口JDBC,它是一种与厂商无关的API，使得Java程序能够与不同厂商的消息组件很好地进行通信。

什么情况下使用

  • 多个项目之间集成
    (1) 跨平台
    (2) 多语言
    (3) 多项目
  • 降低系统间模块的耦合度，解耦
    (1) 软件扩展性
  • 系统前后端隔离
    (1) 前后端隔离，屏蔽高安全区

总结：在客户端与服务器进行通讯时.客户端调用后，必须等待服务对象完成处理返回结果才能继续执行。客户与服务器对象的生命周期紧密耦合,客户进程和服务对象进程都都必须正常运行;如果由于服务对象崩溃或者网络故障导致用户的请求不可达,客户会受到异常。

个人理解在消息并发量大时，可以用到消息队列来缓解服务器压力，避免服务阻塞，挂掉。

准备

  1. 官网下载 ActiveMQ 5 “Classic” 经典版本 https://activemq.apache.org/

  2. 下载完后，进入bin文件夹目录打开cmd命令框

  3. 输入 activemq start 启动消息服务

  4. 然后浏览器输入 http://localhost:8161

  6. 点击see some web domes,可以看到消费者和提供者之间的消息交互，个人理解相当于一个监控统计。

  7. 引入pom文件

  3. 启动时顺序可以反着来一次，这样可以更好理解

    1. 先启动生产者,进行一波键盘后，输入的消息会保存到activemq队列中。此时在启动消费者，消费者会自动将activemq队列中的信息拿出来。它的场景好比：qq聊天时，对方不在线，你发送了十几条信息，只有等到对方上了qq后，才能收到你的发送信息。
    2. 先启动消费者，在启动生产者。它的场景好比：qq聊天时，咱俩都在线，并且互相聊着，你发送一条信息，对方会及时收到。
    3. QUEUE_NAME ： 队列名称，可以理解成一条管道（左进右出||右进左出）。双方必须一致才能收发消息。

  4. 根据自己的理解，聊天机制好比双方同时拥有 一个生产者和一个消费者，这两块功能。生产者负责发送消息到队列，消费者负责从当前队列中取对方消息。同理，对方也拥有这两块功能。并且，我的生产者和对方的消费者必须在同一个队列中，我的消费者和对方的生产者必须在另一个队列中。这样才能做到两个消费者取消息是互不影响，同时又能存入消息。

总结

activemq实现聊天机制的一种方式。

而队列实际上就是缓解高并发时，对服务器造成巨大的压低，导致服务器宕机的一种方式，队列也可以存储用户的请求，不一定是消息。举个例子：在没有做分布式，负载均衡的情况下，几千个用户同时对一台服务器发送请求，假设服务器处理不过来，这时候需要队列将所有的请求进行缓存，然后一条条进行处理。虽然这种说法效率很慢，但总比服务挂掉要强的多…不过一般情况下都会结合分布式进行使用…不过目前为止，这块还没实战过….闲余时间随便看看，记录一下。
